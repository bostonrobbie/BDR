Section 1

System Overview

What the Outreach Command Center does and does not do.

What It Is

A local, browser-based dashboard that sits on top of your file system and a lightweight SQLite database. It consolidates your three outreach channels (LinkedIn, Email, Calls) into one unified workspace where you can research prospects, generate personalized messaging, track every touchpoint, and learn from outcomes over time. It is designed for one user (you) running locally, with no cloud infrastructure required.

What It Does

Research

Agent-powered prospect and company research from Sales Navigator, company websites, news, job postings. Stores structured snapshots so you never research the same company twice.

Draft

Generates copy-paste-ready messages for all 6 touches in your voice, with personalization scores, proof point rotation, and objection pre-mapping. You copy, paste, and send manually.

Track

Logs every touchpoint, reply, status change, and outcome. Links opportunities back to the specific messages and angles that created them. Tracks A/B experiments and personalization scores.

Learn

Analyzes accumulated data to surface what works: which personas convert, which proof points get replies, which openers perform. Generates pre-briefs before each new batch. Gets smarter with every cycle.

What It Does NOT Do

  • Does not send messages. You copy and paste. The system prepares and tracks, you execute.
  • Does not log into platforms autonomously. You provide access via your browser session. The agent reads data from pages you navigate to.
  • Does not make purchasing decisions or financial transactions.
  • Does not store passwords or credentials.
  • Does not violate any platform terms of service. No automated scraping at scale, no bot behavior, no mass messaging.

Core Design Principles

#PrincipleWhat It Means
1Human in the LoopClaude researches and drafts. Rob reviews, edits, and sends. No automated outreach ever.
2Opportunity FocusedEvery metric traces back to opportunities created, not vanity counts like "messages sent."
3Sounds Like RobAll messaging enforces the voice rules in CLAUDE.md. No em dashes, no feature dumps, warm and specific.
4Compounding IntelligenceEvery touchpoint, reply, and outcome feeds the learning loop. Each batch is smarter than the last.
5Token EfficientCache company profiles, reuse research, version message blocks. Don't re-research what you already know.
6Local FirstEverything runs on Rob's machine. No cloud dependencies except Claude. Data never leaves the device.

Email-Specific Rules

  • Subject lines: short, specific, not clickbait. Reference company or role.
  • Body: under 150 words, 2-3 short paragraphs. Mobile-friendly spacing.
  • If no email available, prospect shows in LinkedIn workspace only. Email workspace auto-filters to email-available only.
  • Email touch (Touch 5) can be slightly more direct than InMail since email feels less intrusive.

Email-Specific Agent Behavior

  • Email messages are drafted at the same time as LinkedIn messages (Step 7 of batch build).
  • Subject lines are generated separately and stored in message_drafts.subject_line.
  • Email body can be slightly more direct than InMail per SOP rules.

Call-Specific Agent Behavior

  • Call snippets are pre-generated during batch build (3 lines max per CLAUDE.md rules).
  • Touch 2 and Touch 4 use DIFFERENT proof points and pain angles (variety across channels).
  • Call queue is sorted by: overdue calls first, then priority, then time zone (calls within the prospect's optimal window: 8-11 AM or 3-6 PM local time).
  • Post-call, the Next-Action Agent recommends: if connected + interested → book meeting immediately, suggest times. If voicemail → proceed with Touch 3 on schedule. If connected + not interested → note reason, consider early break-up or adjust pain angle for Touch 3.
  • If outcome = "meeting booked," the system auto-generates a Meeting Prep card from existing research data.

What Gets Fed Back

  • Reply rates by persona type: Adjust mix ratio (more of what converts).
  • Reply rates by proof point: Use winning proof points more, retire underperformers.
  • Reply rates by personalization score: Invest more research time on high-score messages if Score 3 converts better.
  • A/B test results: Lock in winning variants, test new variables.
  • Opportunity attribution: Which specific messages and angles created real pipeline? Double down.
  • Objection patterns: Which objections come up most? Refine pre-loaded responses.
  • Touch effectiveness: Which touch number generates replies? Optimize the sequence.

Token Efficiency Measures

  • Company research caching: Once a company is researched, the snapshot is reused for all contacts at that company.
  • Template-first drafting: Agent starts from persona template, not blank page. Less generation, more personalization.
  • Structured research extraction: Agent fills in a fixed JSON schema, not free-form text. Less tokens, more consistent output.
  • Incremental analysis: Pre-brief only analyzes new data since last run, not all historical data every time.

Build Priority Within Each Phase

Within each phase, build in this order:

  1. Database layer first - tables, migrations, API endpoints. The foundation everything else sits on.

  3. UI third - the dashboard is the interface to the data and agents. It can be basic at first and get polished over time.
  4. Testing last (but continuous) - write tests as you build each component, not after. Unit tests for scoring logic from day one.
